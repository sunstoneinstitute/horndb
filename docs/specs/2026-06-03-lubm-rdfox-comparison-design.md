# Design: Real LUBM-100 materialization comparison (HornDB vs RDFox)

**Date:** 2026-06-03
**Status:** Design — approved for planning
**Tracks:** TASKS.md MEDIUM · _Conformance_ — SPEC-01 harness (RDFox A/B) ([#10](https://github.com/sunstoneinstitute/horndb/issues/10))
**Gate:** BENCHMARKS.md Stage-1 — *LUBM-100 materialization within 3× of RDFox*.

## Problem

The internal HornDB-vs-RDFox comparison harness (commit `339d8b0`) measures
`materialize` against a **synthetic** subClassOf taxonomy, not the literal
benchmark named in the Stage-1 gate. The gate is *LUBM-100 materialization within
3× of RDFox*. We replace the stand-in with a real LUBM workload so the gate is
measured on the benchmark it actually names.

Two blockers, both confirmed this session:

1. **No TBox in a form HornDB can read.** `univ-bench.owl` ships (under
   `<RDFOX_HOME>/examples/data/`) in **OWL Functional Syntax**. Importing it into
   RDFox yields **0 explicit triples** (RDFox parses it as axioms/rules, not RDF),
   and `riot`/`rapper` cannot parse functional syntax. HornDB reads N-Triples and
   so currently has no ontology to reason over.
2. **No matched ruleset.** RDFox ships no OWL 2 RL ruleset (only a 1-line LUBM
   query-completion rule in `additional-rules.txt`). Without a ruleset that fires
   exactly the rules HornDB fires, the two engines do different amounts of work
   and the 3× number is meaningless.

## What we measure (the contract)

Same TBox + same ABox + **same rules** through both engines; compare
materialization wall-clock and **cross-check closure counts**.

- **TBox:** the full real `univ-bench.owl`, converted RDF/XML → N-Triples, kept
  intact — including the 6 complex `EquivalentClasses` definitions
  (e.g. `Student ≡ Person ⊓ ∃takesCourse.Course`).
- **ABox:** LUBM-100 (~13M triples) generated by the Lehigh UBA data generator,
  converted to N-Triples.
- **Rules:** RDFox runs a ruleset *generated from* HornDB's `rules.toml`, so both
  engines fire the same rule set. HornDB lacks `cls-int*`/`cls-svf1`, so the
  6 complex `EquivalentClasses` are inert in HornDB; the generated RDFox ruleset
  excludes those rules too, so they are inert in RDFox as well. Both engines
  therefore ignore the same axioms and the closure should converge.
- **Parity gate:** `HornDB closure count == RDFox closure count` within a known,
  printed eq-axiom offset. Divergence means the ruleset translation is wrong —
  this is the red/green test, not a footnote.

### Licence constraint (non-negotiable)

RDFox numbers are **internal only** (DeWitt clause). All measured RDFox results
land in gitignored `scripts/bench/results/` and `target/bench-rdfox/`. No measured
RDFox number is ever committed — not in `BENCHMARKS.md`, commits, issues, or this
doc. `BENCHMARKS.md` receives **status-only** updates.

## Why HornDB's coverage makes this tractable

LUBM `univ-bench.owl` constructs mapped against HornDB's 49-rule OWL 2 RL subset:

| Construct | Count | HornDB rule(s) | Covered? |
|---|---|---|---|
| `SubClassOf` (atomic) | 36 | `cax-sco`, `scm-sco` | ✅ |
| `SubObjectPropertyOf` | 5 | `prp-spo1`, `scm-spo` | ✅ |
| `InverseObjectProperties` | 2 | `prp-inv1`, `prp-inv2` | ✅ |
| `ObjectPropertyDomain`/`Range` | 39 | `prp-dom`, `prp-rng`, `scm-dom*`, `scm-rng*` | ✅ |
| `TransitiveObjectProperty` (`subOrganizationOf`) | 1 | `prp-trp` | ✅ |
| `DataPropertyDomain` | 4 | `prp-dom` | ✅ |
| `EquivalentClasses` w/ `∃P.C ⊓ Person` | 6 | `cls-int1` + `cls-svf1` | ❌ **not in subset** |

Divergence is concentrated entirely in the 6 complex `EquivalentClasses`. By
generating the RDFox ruleset from `rules.toml`, that divergence is removed at the
ruleset level rather than papered over.

## Components

### 1. Ruleset generator — `scripts/bench/gen_ruleset.py`

Reads `crates/owlrl/rules.toml`; emits `scripts/bench/rules/owl2rl-horndb-subset.dlog`
in RDFox Datalog. The TOML is a uniform table — each `[[rule]]` has
`body = [{s,p,o}, …]` and `head = {s,p,o}` — so the transform is mechanical:

```
head=:{s,p,o}, body=[b0,b1]   →   [head.s, head.p, head.o] :- [b0…], [b1…] .
```

Classification is driven by each rule's **head**:

- **Include** every rule whose head is a derivable triple — i.e. *additive*
  rules. This includes the `delegate = "closure"` rules (transitivity of
  `subClassOf`/`subPropertyOf`/transitive properties, `sameAs`
  reflexivity/symmetry/transitivity) — HornDB computes these via GraphBLAS,
  RDFox via native recursion; they must be present or the closures will not
  match. It **also** includes rules that derive `owl:sameAs` from property
  characteristics (`prp-fp`, `prp-ifp` — functional / inverse-functional
  property → objects are `sameAs`); these are additive, not inconsistency rules.
- **Omit** rules whose head asserts an inconsistency / clash rather than a
  derivable triple (e.g. `cax-dw`, `cls-com`, `prp-npa1`, `prp-npa2`, `prp-irp`,
  `prp-asyp`, `prp-pdw`, `eq-diff1`). These derive no triples, so they cannot
  affect the closure count, and LUBM's ABox does not exercise them anyway. The
  generator classifies from the head shape (not a hard-coded id list) and emits a
  header comment listing every rule it omitted and why.
- Prefix declarations (`rdf:`, `rdfs:`, `owl:`) emitted from the same prefix set
  `rules.toml` uses, matching the existing `owl2rl-core.dlog` syntax.

The closure-count parity gate validates the translation end-to-end; we do not
hand-verify 49 rules.

### 2. LUBM data pipeline — `scripts/bench/gen_lubm.sh --universities N`

1. Fetch the UBA1.7 generator (Java) and the canonical **RDF/XML**
   `univ-bench.owl` from Lehigh. The RDF/XML ontology variant solves blocker (1)
   *and* is the form UBA needs. (UBA bundles a `univ-bench.owl`; if it is the
   RDF/XML variant, reuse it instead of a separate fetch.)
2. `java … edu.lehigh.swat.bench.uba.Generator -univ N -seed 0 -onto
   http://swat.cse.lehigh.edu/onto/univ-bench.owl` → deterministic per-university
   RDF/XML files.
3. `riot --syntax=RDFXML --output=NT` batch-convert the university files → one
   `abox.nt`; convert the RDF/XML ontology → `tbox.nt`. Sanity-check
   (`grep -c subClassOf tbox.nt` etc.).
4. **Validate at N=1 first.** Run the whole pipeline + comparison at N=1 and
   cross-check the generated `abox.nt` against the on-disk `lubm1.ttl`
   (already N-Triples-shaped) before committing to the N=100 (~13M-triple,
   multi-GB) run. Then run N=100 for the gate number.

All generated artifacts (generator jar, RDF/XML files, `.nt` files) are
gitignored under `target/bench-rdfox/`.

### 3. Harness mode — `scripts/bench/compare-rdfox.sh --lubm N`

- Ensures the LUBM data exists (invokes `gen_lubm.sh --universities N` if not)
  and the ruleset is generated (invokes `gen_ruleset.py` if stale).
- Feeds `tbox.nt + abox.nt` to `horndb-bench materialize` (the runner already
  accepts multiple `--data`).
- Feeds the same `tbox.nt + abox.nt` + generated `owl2rl-horndb-subset.dlog` to
  RDFox, reusing the existing reason-isolation (import data first, rules last).
- Asserts the **closure-count cross-check** (HornDB `materialized_len` vs RDFox
  count, within the printed eq-axiom offset). This is a hard gate, not a print.
- **Wall-clock cap** (configurable, e.g. `--cap-seconds`): if HornDB's nested-loop
  `RuleFiringBackend` does not finish within the cap, the harness records
  "did not complete within <cap>" as a **valid Stage-1 finding** (see Risks).
- Stores results under gitignored `scripts/bench/results/`.

## Risks

- **HornDB nested-loop backend on 13M triples.** `horndb-bench materialize` uses
  `RuleFiringBackend` (nested-loop), not the GraphBLAS closure path. On 13M
  triples it may be very slow or OOM. The wall-clock cap turns this into a
  recorded outcome rather than a hang. A result of "HornDB > 3× / did not finish"
  is a real, useful Stage-1 signal — not a harness bug.
- **Java toolchain.** UBA is Java. The pipeline checks for `java` and documents
  the requirement; absence is a clear, early failure.
- **Disk pressure.** Multi-GB N-Triples for LUBM-100. Gitignored, but real local
  pressure; documented in `scripts/bench/README.md`.
- **Ontology variant mismatch.** If the fetched RDF/XML `univ-bench.owl` differs
  from the functional-syntax copy in axiom content, closure counts shift. The
  N=1 validation against `lubm1.ttl` catches gross mismatches early.

## Out of scope

- Publishing any RDFox number (legal review required; SPEC-01 Risks).
- LUBM-8000 / Stage-2 scale.
- GraphBLAS closure-path comparison (this measures the rule-firing backend, which
  is what the synthetic harness already exercises).
- Changes to HornDB's rule set (e.g. adding `cls-int`/`cls-svf1`) — a separate
  decision; this design deliberately compares the *current* subset.

## Acceptance

1. `gen_ruleset.py` produces `owl2rl-horndb-subset.dlog` from `rules.toml`; the
   generated file lists omitted inconsistency rules in its header.
2. `gen_lubm.sh --universities 1` produces `tbox.nt` + `abox.nt`; `abox.nt`
   cross-checks against the on-disk `lubm1.ttl`.
3. `compare-rdfox.sh --lubm 1` runs both engines and the closure-count parity
   gate passes (within the printed eq-axiom offset).
4. `compare-rdfox.sh --lubm 100` runs to a recorded outcome (a timing within the
   cap **or** an explicit "did not complete within cap"), with the parity gate
   evaluated.
5. `BENCHMARKS.md` Stage-1 row updated **status-only** (no RDFox number);
   `TASKS.md` #10 reflects the new state; pre-push gate (clippy + build) stays
   green.
